# 核心类图设计

> 基于 P1 需求规格说明书和 P2 架构设计（分层架构 + MVC 表现层，ADR-001），设计"青隅"校园互助服务平台的核心类图。遵循 P2 定义的 8 大模块划分（用户/树洞/找搭子/恋爱互助/通知/审核/管理员/反馈）。

---

## 一、系统类图总览

### 1.1 分层架构与类归属（对齐 P2 四层架构）

本系统类图严格遵循 P2 ADR-001 确定的分层架构 + MVC 表现层混合模式，各类按职责划分至不同层级：

| 层级 | 包含类 | 职责 |
|------|-------|------|
| 表示层 (Controller) | UserController, TreeHoleController, PartnerController, LoveController, AdminController | 接收 HTTP 请求，参数校验，调用业务层 |
| 业务层 (Service) | UserService, TreeHoleService, PartnerService, LoveService, ReviewService, NotificationService | 核心业务逻辑，事务管理 |
| 数据访问层 (Repository) | UserRepository, TreeHoleRepository, PartnerRepository, LoveRepository, ReviewRepository | 数据库 CRUD 操作 |
| 领域模型层 (Model) | User, Admin, TreeHolePost, TreeHoleComment, PartnerRequest, PartnerMatch, LoveProfile, LoveMatch, Notification, ReviewRecord, Feedback | 数据实体，领域对象 |
| 工具/基础设施层 | SecurityUtil, ImageValidator, SensitiveWordFilter, SmsService, IdCardVerifier | 横切关注点，基础设施 |

> **说明：** P2 采用前后端分离（ADR-003），前端为 Vue 3，通过 RESTful API 调用后端。Controller 层即 MVC 中的 C，负责接收前端请求并返回 JSON 响应。事件驱动机制（观察者模式）仅用于通知和日志的局部解耦，并非系统主架构。

## 三、设计模式应用

### 3.1 观察者模式（Observer Pattern）—— 消息通知系统

> **对齐 P2：** P2 架构定位中明确采用"局部事件驱动（通知、日志）"的扩展机制。观察者模式正是这一架构决策在详细设计层的落地实现。

**应用场景：** 当业务事件发生时（树洞被点赞/评论、搭子匹配成功、心动匹配等），需要向相关用户推送通知。

**为什么使用：**
- 业务模块（TreeHoleService、PartnerService、LoveService）不应直接依赖通知模块的具体实现，否则违反 DIP 原则
- 一个业务事件可能触发多种通知渠道（站内消息、短信、Push），观察者模式可以让各通知渠道独立订阅
- 新增通知渠道时无需修改业务代码（符合 OCP）

**类结构：**
```
┌─────────────────┐         ┌─────────────────────────┐
│ <<interface>>   │         │  <<interface>>          │
│ EventPublisher  │         │  EventListener          │
├─────────────────┤         ├─────────────────────────┤
│ + subscribe()   │         │ + onEvent(event): void  │
│ + unsubscribe() │         └───────────┬─────────────┘
│ + notify()      │                     △
└────────┬────────┘       ┌─────────────┼──────────────┐
         △                │             │              │
         │          ┌─────┴────┐  ┌─────┴────┐  ┌─────┴─────┐
┌────────┴───────┐  │InAppNotif│  │SmsNotif  │  │PushNotif  │
│EventBus        │  │Listener  │  │Listener  │  │Listener   │
├────────────────┤  └──────────┘  └──────────┘  └───────────┘
│ - listeners    │
└────────────────┘
```

**如果不用这个模式：** 业务代码中会直接调用 `notificationService.send()`，导致业务逻辑与通知逻辑紧耦合，每新增一种通知方式就需修改业务代码。

### 3.2 策略模式（Strategy Pattern）—— 内容审核系统

**应用场景：** 不同类型的内容（树洞文字、树洞图片、搭子需求、交友照片）需要不同的审核策略，且审核方式可能扩展（自动审核、人工审核、AI审核）。

**为什么使用：**
- 不同内容类型的审核规则不同：文字审核用敏感词库，图片审核用第三方 API
- 未来可能引入 AI 语义审核，需要能灵活替换审核策略
- 避免在 ReviewService 中使用大量 if-else 判断内容类型

**类结构：**
```
┌─────────────────────┐
│  <<interface>>      │
│  ReviewStrategy     │
├─────────────────────┤
│ + review(content):  │
│   ReviewResult      │
└──────────┬──────────┘
           △
   ┌───────┼───────────────┐
   │       │               │
┌──┴───────┴──┐  ┌────────┴──────┐  ┌──────────────┐
│TextReview   │  │ImageReview    │  │AIReview      │
│Strategy     │  │Strategy       │  │Strategy      │
├─────────────┤  ├───────────────┤  ├──────────────┤
│- wordBank   │  │- apiClient    │  │- aiClient    │
├─────────────┤  ├───────────────┤  ├──────────────┤
│+ review()   │  │+ review()     │  │+ review()    │
└─────────────┘  └───────────────┘  └──────────────┘

┌──────────────────────┐
│  ReviewContext       │
├──────────────────────┤
│ - strategy           │
├──────────────────────┤
│ + setStrategy()      │
│ + executeReview()    │
└──────────────────────┘
```

**如果不用这个模式：** ReviewService 中会充满 `if (contentType == TEXT)` / `else if (contentType == IMAGE)` 的条件分支，新增审核类型时需要修改已有代码，违反 OCP。

## 四、SOLID 原则在类图中的体现

| 原则 | 设计体现 |
|------|---------|
| S - 单一职责 | TreeHolePost 仅负责树洞动态的 CRUD，不涉及通知、审核；ReviewRecord 只负责审核记录，不涉及业务逻辑 |
| O - 开闭原则 | ReviewStrategy 接口使审核策略可扩展；EventListener 接口使通知渠道可扩展 |
| L - 里氏替换 | NormalUser 和 Admin 均可替代 BaseUser 使用，子类行为不改变父类契约 |
| I - 接口隔离 | IAuthenticatable 只定义认证相关方法；ReviewStrategy 只定义审核方法——接口小而专 |
| D - 依赖倒转 | EventBus 依赖 EventListener 抽象接口，而非具体的 InAppNotifListener；ReviewContext 依赖 ReviewStrategy 抽象，而非具体策略类 |
