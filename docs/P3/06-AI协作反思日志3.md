## 2. AI 助力分析（Q1）

### AI 在本阶段表现出色的方面：

1. **结构化输出能力强**：AI 能够快速生成类图、ER 图、API 文档的结构化描述，在格式编排上节省了大量时间。例如，建表 SQL 的字段类型选择、注释规范等细节，AI 的输出质量达到 4 分。

2. **覆盖面完整**：AI 在生成类图和 API 时不会遗漏模块，能够确保所有 P0 需求都有对应的设计元素覆盖。这对于大模块项目尤为重要——人工设计容易在交叉模块间遗漏细节。

3. **SOLID 原则的"反向检查"能力强**：当给定 AI 自己的设计时，AI 在识别 SOLID 违规方面表现很好（4分）。AI 对设计原则的知识储备充分，能够准确识别 LSP 违反和接口隔离问题。

4. **索引设计建议合理**：AI 能够根据查询场景给出索引设计建议，包括联合索引的最左前缀原则、唯一索引的业务意义等。

### AI 节省的核心工时：

- 类图结构化描述编写：节省约 1 小时
- SQL DDL 编写（18 张表）：节省约 1.5 小时
- API 文档格式编排：节省约 1 小时
- **总计节省约 3.5 小时 / 团队 6 小时中的 58%**

## 3. AI 误导分析（Q2）

### SOLID 检查中发现的 AI 设计问题数量：**7 处**

| 严重程度 | 数量 | 代表问题 |
|---------|------|---------|
| 严重 | 4 处 | TreeHoleService 职责过多、NormalUser 违反 LSP、胖接口 IContentManager、Service 直接依赖 MySQLDatabase |
| 中等 | 3 处 | 审核逻辑硬编码、User 类隐私属性散落、SmsService 依赖具体 SDK |

### 最严重的设计问题是：

**TreeHoleService 同时承担了业务逻辑、审核逻辑、通知推送三个职责。** 这个问题说明 AI 在"职责划分"上缺乏工程直觉。AI 倾向于将功能相关的代码"堆"进同一个类，而不是分析"变更原因"来做职责分离。如果按 AI 的原始设计实现，TreeHoleService 会在 2-3 次迭代后膨胀到 500+ 行，成为维护噩梦。

### AI 容易出现的系统性偏差：

1. **"胖类"倾向**：AI 偏好创建大而全的类（God Class），缺乏对单一职责的敏感度
2. **继承优先于组合**：AI 在没有充分验证 LSP 的情况下倾向于使用继承
3. **忽视依赖方向**：AI 不擅长主动应用 DIP，默认从具体到具体
4. **安全性盲区**：AI 在 API 设计中会忘记认证标注、敏感信息脱敏、验证码防护等安全措施

### 应对策略：

- SOLID 原则不应作为设计输入（让 AI 自由设计），而应作为设计审查工具（人工检查 AI 设计）
- 安全相关设计（密码存储、数据加密、脱敏规则）必须人工定义，AI 协助执行
- 类职责划分的最终决定权必须在人类工程师手中
- 延续 P2 的经验：AI 在架构决策中偏向"技术最优解"（如引入 Redis、微服务），本阶段在详细设计中同样发现 AI 倾向于过度设计（如胖类、胖接口），需人工做减法

## 4. Prompt 改进计划（Q3）

### 上阶段改进措施的验证结果：

- **P2 改进①**："Prompt 中明确要求结合 4 人团队/10 周周期/校园小规模场景" → **部分有效**。本阶段 API/类图设计的 Prompt 中未显式注入这些约束，导致 AI 仍在性能优化部分推荐 Redis、Elasticsearch 等超出当前架构范围的方案。**反思：** P2 的约束型 Prompt 经验需在每个阶段的 Prompt 中复用。
- **P2 改进②**："要求 AI 输出需包含工程落地成本评估" → **未在本阶段 Prompt 中贯彻**。AI 生成的设计模式选择未评估引入复杂度 vs 收益。后续需在 Prompt 中强制要求。
- **P2 改进③**："禁止推荐超出校园小规模场景的过度技术方案" → **验证有效**。本阶段在数据库设计 Prompt 中明确要求"单一 MySQL 数据库，不使用分库分表"，AI 输出未越界。

### 本阶段新的改进措施：

> 继承 P2 的"约束型 Prompt + 禁止项"策略，并针对详细设计阶段特点做增强：

1. **在设计 Prompt 中显式注入架构约束（继承 P2 经验）**：
   - 原 Prompt："请设计系统类图" → AI 违反 7 处 SOLID
   - 新 Prompt："请设计系统类图。架构约束：单体分层架构（非微服务）、MySQL 8.0 单一数据库、暂不使用 Redis/消息队列。设计要求：每个类遵循单一职责原则，业务层通过接口依赖数据层。" → 预期将违规数降至 2-3 处

2. **API 设计 Prompt 增加安全清单（P2 ADR-003 延伸）**：
   - 在 API 设计 Prompt 中追加："请为每个接口标注是否需 JWT 认证、敏感字段需脱敏、需包含图形验证码保护"
   - 预期将 API 安全性相关问题从 5 处降至 1-2 处

3. **采用"AI 生成 + 人工 SOLID 审查"的两阶段工作流**：
   - 阶段 1：AI 快速生成初版设计（利用覆盖面优势）
   - 阶段 2：人工按 S→D→I→L→O 顺序逐条审查（利用判断力优势）
   - 这一工作流已成为团队的固定实践

4. **P2→P3 延续性改进**：将每个阶段验证有效的 Prompt 策略记录为团队模板，避免在每个阶段从零开始调优 Prompt。例如 P2 的"禁止越界技术方案"约束应在后续所有阶段的 Prompt 中默认包含。

## 5. 本阶段的核心工程判断

### 决策内容：

选择"观察者模式"而非"直接在业务方法中调用通知服务"来实现消息通知系统。该决策与 P2 架构定位中"局部事件驱动（通知、日志）"的扩展机制一致——在保持整体分层架构的前提下，对通知场景做局部解耦。

### 为什么 AI 无法做这个决策：

AI 能够识别出两种方案，但无法判断**何时**值得引入设计模式的额外复杂度。这个决策依赖于对以下不可量化因素的判断：

1. **团队熟悉度**：团队对观察者模式有实践经验，引入成本低
2. **变化预期**：根据需求分析，通知渠道在 v1.0~v2.0 之间很可能会从"站内通知"扩展到"Push + 短信 + 微信"，观察者模式可以无痛扩展
3. **团队规模**：4 人团队，模块分工明确，观察者模式的解耦效果利于并行开发
4. **与 P2 架构的一致性**：P2 已明确"局部事件驱动"的扩展定位，观察者模式恰好是这一架构决策在详细设计层的落地

AI 无法评估"组织因素"和"对未来变化的判断"——这些需要基于团队经验的工程直觉。同样，P2 中"放弃微服务、暂不引入 Redis"等关键决策，也是 AI 无法基于纯粹技术分析做出的工程权衡。
