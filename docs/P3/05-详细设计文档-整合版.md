## 一、设计概述

### 1.1 设计目标

本文档整合"青隅"校园互助服务平台的详细设计成果，覆盖：
- **类图设计**：核心业务类的属性、方法、关系
- **SOLID 符合性**：确保设计符合面向对象设计原则
- **API 规范**：RESTful 风格接口定义
- **数据库设计**：ER 图、建表 SQL、索引策略

### 1.2 设计依据

- P1 需求规格说明书（平台包含树洞、找搭子、恋爱互助三大核心模块）
- P2 架构设计（分层架构 + MVC 表现层，ADR-001；前后端分离 + RESTful + JWT，ADR-003）
- P2 模块划分（8 大核心模块：用户/树洞/找搭子/恋爱互助/通知/审核/管理员/反馈）
- P2 技术选型（MySQL 8.0 单一数据库 ADR-002，暂不引入 Redis ADR-005，本地存储）
- IEEE 830 需求规格标准
- SOLID 面向对象设计原则

### 1.3 技术栈

| 层级 | 技术选型 |
|------|---------|
| 后端框架 | Spring Boot（本地 JAR 部署） |
| 前端 (Web) | Vue 3 |
| 移动端 | Flutter |
| 数据库 | MySQL 8.0（单一数据库，ADR-002） |
| 文件存储 | 本地存储 |
| 认证 | JWT (JSON Web Token)，ADR-003 |
| 密码加密 | BCrypt |
| 敏感数据加密 | AES-256-CBC |
| 缓存 | 暂不引入 Redis（ADR-005：后期按需评估） |
| 通知异步化 | 后期可选引入消息队列（当前阶段本地同步处理） |

## 三、核心类图设计

### 3.1 类层次结构

**用户继承体系：**
```
BaseUser (抽象基类)
├── NormalUser (普通用户)
│   ├── 关联 → PrivacySettings (1:1 组合)
│   ├── 关联 → UserCert (1:1 认证信息)
│   └── 实现 → IAuthenticatable (认证接口)
└── Admin (管理员)
    └── 关联 → ReviewRecord (1:N 审核记录)
```

### 3.2 核心业务类

| 模块 | 核心类 | 职责 | 关键关系 |
|------|-------|------|---------|
| 用户 | BaseUser, NormalUser, Admin | 用户认证、资料管理 | Admin 继承 BaseUser |
| 树洞 | TreeHolePost, TreeHoleComment, TreeHoleLike | 匿名动态的发布与互动 | Post 1:N Comment, Post 1:N Like |
| 找搭子 | PartnerRequest, PartnerMatch, PartnerChat, PartnerReview | 搭子需求发布、匹配、聊天、评价 | Request 1:N Match, Match 1:N Chat |
| 恋爱互助 | LoveProfile, LoveRequest, LoveMatch, LoveChat | 交友资料、心动匹配、聊天 | User 1:1 Profile, Profile 1:1 Request |
| 通知 | Notification, MessageSetting | 消息推送、通知设置 | User 1:N Notification |
| 审核 | ReviewRecord, SensitiveWord | 内容审核、违规词库 | Admin 1:N ReviewRecord |
| 反馈 | Feedback | 用户反馈建议 | User 1:N Feedback |

### 3.3 关系类型定义

| 关系 | 符号 | 示例 |
|------|------|------|
| 继承 | △ (三角形箭头) | NormalUser → BaseUser |
| 实现 | △ 虚线 | NormalUser → IAuthenticatable |
| 关联 | ── (实线) | NormalUser ── TreeHolePost |
| 聚合 | ◇ (空心菱形) | NormalUser ◇── PrivacySettings |
| 组合 | ◆ (实心菱形) | 未使用（避免过度设计） |
| 依赖 | ┈┈┈ (虚线箭头) | Controller ┈┈┈ Service |

## 四、SOLID 原则应用与检查

### 4.1 AI 设计违规统计

| 原则 | 违规数 | 严重程度分布 |
|------|--------|------------|
| S - 单一职责 | 2 | 严重 ×1, 中等 ×1 |
| O - 开闭原则 | 1 | 中等 ×1 |
| L - 里氏替换 | 1 | 严重 ×1 |
| I - 接口隔离 | 1 | 严重 ×1 |
| D - 依赖倒转 | 2 | 严重 ×1, 中等 ×1 |
| **合计** | **7** | 严重 ×4, 中等 ×3 |

### 4.2 关键修正措施

1. **单一职责**：`TreeHoleService` 拆分为 `TreeHoleService` + `ContentReviewService` + `NotificationService`
2. **开闭原则**：引入策略模式，审核策略可扩展不需修改原有代码
3. **里氏替换**：`NormalUser.login()` 不再增强前置条件，身份校验移至 `certify()` 阶段
4. **接口隔离**：`IContentManager`（7个方法）拆分为 `IPublisher` + `IReviewer` + `INotifiable`
5. **依赖倒转**：Service 通过构造函数注入 `Repository` 接口，而非直接依赖 `MySQLDatabase`
6. **依赖倒转**：`SmsService` 依赖 `ISmsProvider` 接口，具体提供商作为实现注入

### 4.3 SOLID 设计自查结论

经过审查和修正，当前设计已满足全部 5 条 SOLID 原则，各原则的符合性如下：

| 原则 | 状态 | 说明 |
|------|------|------|
| S | ✅ 符合 | 每个类职责单一，TreeHolePost 只管动态 CRUD |
| O | ✅ 符合 | 审核策略、通知渠道均可扩展 |
| L | ✅ 符合 | NormalUser/Admin 可安全替换 BaseUser |
| I | ✅ 符合 | 接口小而专（IAuthenticable、ReviewStrategy 等） |
| D | ✅ 符合 | 业务层依赖 Repository 抽象接口 |

## 五、设计模式应用

### 5.1 观察者模式 — 消息通知

| 项目 | 说明 |
|------|------|
| 应用场景 | 当树洞被点赞/评论、搭子匹配成功、心动匹配等事件发生时的通知推送 |
| 参与类 | EventBus（发布者）、EventListener 接口、InAppNotifListener / SmsNotifListener / PushNotifListener（订阅者） |
| 收益 | 解耦业务逻辑与通知逻辑；新增通知渠道无需修改业务代码；支持运行时注册/注销监听器 |

### 5.2 策略模式 — 内容审核

| 项目 | 说明 |
|------|------|
| 应用场景 | 不同类型内容的审核（文字敏感词、图片审核、AI 语义审核） |
| 参与类 | ReviewStrategy 接口、TextReviewStrategy / ImageReviewStrategy / AIReviewStrategy、ReviewContext |
| 收益 | 避免大量 if-else 分支；新增审核策略无需修改 ReviewService；审核策略可动态切换 |

## 七、数据库设计

### 7.1 ER 图核心实体

| 模块 | 表名 | 说明 |
|------|------|------|
| 用户 | tb_user, tb_user_cert, tb_privacy_setting, tb_admin | 用户基础信息、认证、隐私、管理员 |
| 树洞 | tb_treehole_post, tb_treehole_comment, tb_treehole_like | 动态、评论、点赞 |
| 找搭子 | tb_partner_req, tb_partner_match, tb_partner_chat, tb_partner_review | 需求、匹配、聊天、评价 |
| 恋爱互助 | tb_love_profile, tb_love_req, tb_love_match, tb_love_chat, tb_love_review | 资料、需求、匹配、聊天 |
| 系统 | tb_notification, tb_message_setting, tb_review_record, tb_sensitive_word, tb_feedback | 通知、设置、审核、词库、反馈 |

**共 18 张表。**

### 7.2 关键索引策略

- **tb_treehole_post**: 联合索引 `idx_category_status(category, status)` 加速首页分类浏览
- **tb_treehole_post**: 联合索引 `idx_status_created(status, created_at)` 加速按时间排序
- **tb_partner_req**: FULLTEXT 索引 `ft_description` 加速关键词搜索
- **tb_notification**: 联合索引 `idx_user_read_time(user_id, is_read, created_at)` 加速未读通知查询
- **tb_treehole_like**: 唯一索引 `uk_post_user(post_id, user_id)` 防止重复点赞

### 7.3 数据安全

| 安全措施 | 说明 |
|---------|------|
| 密码存储 | BCrypt 哈希（cost=10），不可逆 |
| 敏感字段 | 手机号、学号、姓名、身份证号使用 AES-256-CBC 加密 |
| 展示脱敏 | 手机号显示为 `138****8000` |
| 传输加密 | 全程 HTTPS |

## 八、安全设计

### 8.1 认证与授权

| 机制 | 说明 |
|------|------|
| JWT Token | 登录后签发，有效期 2 小时（rememberMe 7 天） |
| Token 刷新 | 通过 Refresh Token 续期 |
| 登录保护 | 连续 3 次失败锁定 15 分钟，图形验证码 |
| 实名认证 | 学号 + 姓名 + 身份证号 + 人脸识别 |

### 8.2 内容安全

| 层次 | 措施 |
|------|------|
| 事前 | 敏感词库自动过滤（Trie/AC 自动机） |
| 事前 | 第三方图片审核 API |
| 事后 | 人工审核队列（疑似违规内容） |
| 事后 | 用户举报机制 |

### 8.3 隐私保护

| 场景 | 措施 |
|------|------|
| 树洞 | 全程匿名，系统自动生成匿名昵称 |
| 找搭子 | 匹配确认前不展示真人隐私信息 |
| 恋爱交友 | 匹配确认前不展示手机号、学号 |
| 数据存储 | 敏感字段加密存储 |
| 注销 | 注销后 7 天彻底删除数据 |

### 8.4 部署与基础设施（对齐 P2 ADR 决策）

| 决策项 | P2 ADR 结论 | P3 跟进 |
|--------|------------|---------|
| 架构模式 | ADR-001：分层架构 + MVC 表现层 | 类图/模块划分遵循四层结构 |
| 数据库 | ADR-002：MySQL 8.0 单一数据库 | ER 图/建表 SQL 均基于单库设计 |
| 前后端分离 | ADR-003：前后端分离 + RESTful + JWT | API 规范采用 RESTful 风格，JWT 认证 |
| 微服务 | ADR-004：废弃，维持单体 | 所有模块为进程内调用，类间关系为本地依赖 |
| 缓存 | ADR-005：暂不引入 Redis | 性能优化方案标注为"后期按需评估" |
| 文件存储 | 本地存储（最简架构） | 图片/文件使用本地路径存储 |
| 部署 | 本地 JAR 包 | 单服务器部署，最小化外部依赖 |
