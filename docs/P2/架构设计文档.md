# 架构设计文档
## 1. 架构概览（含架构图）
### 1.1 项目背景
本项目为**校园互助服务平台（青隅）**，面向高校大学生，提供树洞倾诉、找搭子、恋爱互助三大核心服务。
团队规模：4 人
开发周期：10 周
最终架构：**分层架构 + MVC 表现层** 混合模式

### 1.2 架构定位
- 主体架构：**分层架构（Layered Architecture）**
- 表现层组织：**MVC 模式**
- 扩展机制：局部事件驱动（通知、日志）
- 暂不采用：微服务、分布式、事件驱动主架构

### 1.3 整体架构图
```
┌─────────────────────────────────────────────────────┐
｜                     前端层（Vue3）                  ｜
└───────────────────────────┬─────────────────────────┘
                            │
┌───────────────────────────▼─────────────────────────┐
｜                    控制层（Controller）             ｜
｜ （用户、树洞、搭子、恋爱、通知、审核、反馈、管理员）    ｜
└───────────────────────────┬─────────────────────────┘
                            │
┌───────────────────────────▼─────────────────────────┐
｜                     业务层（Service）               ｜
｜          核心逻辑、流程编排、规则校验、权限控制       ｜
└───────────────────────────┬─────────────────────────┘
                            │
┌───────────────────────────▼─────────────────────────┐
｜                  数据访问层（Repository）           ｜
｜              数据库 CRUD、事务、数据封装             ｜
└───────────────────────────┬─────────────────────────┘
                            │
┌───────────────────────────▼─────────────────────────┐
｜                 基础设施层（Infrastructure）        ｜
｜  MySQL、Redis、文件存储、第三方接口、审核、短信、推送  ｜
└─────────────────────────────────────────────────────┘
```

### 1.4 架构选择理由
1. **符合 4 人团队 + 10 周周期**，学习成本低、风险最低。
2. **结构清晰、易于分工**，模块边界明确。
3. **完全支撑所有需求**：用户、树洞、找搭子、恋爱互助、审核、通知、反馈、后台管理。
4. **易开发、易调试、易测试**，非常适合课程项目。
5. **性能足够**，本地调用无分布式开销，满足校园规模。
6. **可维护性强**，层级分明，便于后续扩展。

## 2. 模块划分说明
### 2.1 总体模块划分
本平台共 **8 大核心业务模块**：

1. **用户模块（User）**
2. **树洞模块（TreeHole）**
3. **找搭子模块（Partner）**
4. **恋爱互助模块（Love）**
5. **消息通知模块（Notify）**
6. **内容审核模块（Audit）**
7. **管理员模块（Admin）**
8. **反馈建议模块（Feedback）**

---

### 2.2 模块职责明细
#### （1）用户模块
- 用户注册、登录、短信验证
- 学号认证、实名认证
- 个人资料、隐私设置、头像、兴趣标签
- 账号安全、密码找回、注销
- 提供用户信息查询接口给其他模块

#### （2）树洞模块
- 匿名发布动态（文字 + 图片）
- 匿名浏览、点赞、评论、回复
- 分类、搜索、热度排序
- 动态管理（编辑、删除、可见范围）
- 内容审核接入
- 心理健康专栏展示

#### （3）找搭子模块
- 发布搭子需求（学习/运动/吃饭/考研/出行等）
- 匹配条件：年级、专业、性别、时间、地点
- 自动匹配 + 主动申请
- 搭子聊天、群聊、评价体系
- 搭子关系管理（待确认/已建立/已结束）

#### （4）恋爱互助模块
- 交友资料完善（照片、身高、星座、择偶要求）
- 交友需求发布、匹配、推荐
- 聊天、语音、礼物互动
- 交友安全提醒、违规举报
- 隐私保护、敏感信息隐藏

#### （5）消息通知模块
- 系统公告、审核结果通知
- 点赞、评论、回复、匹配通知
- 消息免打扰、开关配置
- 多端同步

#### （6）内容审核模块
- 文字违规词过滤
- 图片自动审核
- 人工审核工作台
- 审核记录、内容风险控制

#### （7）管理员模块
- 用户管理、封禁、解封
- 内容管理、删帖、置顶
- 违规词库管理
- 数据统计、日志、备份
- 反馈处理

#### （8）反馈建议模块
- 用户提交反馈、上传截图
- 处理进度查询
- 管理员回复与记录

---

### 2.3 模块依赖关系
```
用户模块
   ↑   ↑   ↑   ↑
   │   │   │   │
树洞 搭子 恋爱 反馈
   │   │   │
   └───┼───┘
       │
通知模块 ← 审核模块
       │
管理员模块
```

### 2.4 模块接口规范
- 调用方式：RESTful API + 本地方法调用
- 数据格式：JSON + DTO
- 统一返回格式：code / msg / data
- 所有跨模块调用必须通过接口，不直接依赖实现类

## 3. 技术选型说明
| 层次 | 技术选型 | 选择理由 |
|------|----------|----------|
| 前端框架 | Vue 3 | 简单高效、生态成熟、适合快速开发 |
| 后端框架 | Spring Boot | 完美支持分层架构，注解开发、稳定易学 |

| 部署方式 | 本地 JAR 包 | 环境简单、交付快速、10 周可完成 |

# ADR-001：采用分层架构 + MVC 表现层作为系统主架构
## 状态
已接受
## 背景
项目为校园互助服务平台（青隅），4 人团队、10 周周期，需支持用户、树洞、找搭子、恋爱互助、内容审核、消息通知等全业务功能，需从多种架构中选定主架构。
## 决策
我们决定采用**分层架构为主体，MVC 作为表现层组织方式**的混合架构。（diff）
## 理由
1. 开发复杂度低、学习成本低，适合小团队快速交付
2. 层级清晰、便于分工、调试简单、风险可控
3. 完全匹配 10 周周期，可保证按期稳定交付
4. 可完整支撑树洞、搭子、恋爱互助三大核心场景（diff）
## 后果（diff）
- 正面影响：开发效率高、结构清晰、易维护、便于答辩
- 负面影响：扩展性不如微服务，仅支持整体系统扩容
- 需要关注的风险：业务膨胀后可能出现逻辑耦合，需通过 DTO 与接口规范约束
## AI 辅助记录
- **AI 初稿内容摘要**：推荐使用分层架构，理由为简单、稳定、易开发。
- **人工修订内容（diff）**：
  - **新增**：MVC 表现层定位
  - **新增**：混合架构模式
  - **新增**：结合树洞、搭子、恋爱互助业务场景
  - **新增**：质量属性（可维护性、安全性、性能）权衡
- **修订理由**：AI 输出过于通用，未结合项目实际业务与团队约束。

# ADR-002：采用 MySQL 8.0 作为项目唯一数据库
## 状态
已接受
## 背景
平台需存储用户、树洞、搭子、恋爱交友、消息、评论、审核记录等数据，需确定数据库类型与数量。
## 决策
我们决定采用 **MySQL 8.0 单一数据库** 存储全部业务数据，不使用分库分表、NoSQL 或多数据源。（diff）
## 理由（diff）
1. 校园项目数据量小，单库完全满足需求
2. 团队学习MySQL成本较低
3. 无需处理分布式事务，架构简单稳定
4. 降低开发、测试、部署复杂度，保障 10 周可交付
## 后果
- 正面影响：架构简单、无分布式问题、开发快速稳定
- 负面影响：面对超大规模数据扩展性有限
- 需要关注的风险：表数量增多后可能变慢，将通过索引与分页优化
## AI 辅助记录
- **AI 初稿内容摘要**：建议使用 MySQL，理由为流行、轻量、稳定。
- **人工修订内容（diff）**：
  - **明确拒绝**：多数据源、MongoDB 等 NoSQL 方案
  - **新增**：结合校园数据规模、周期紧张、团队技术约束进行论证
- **修订理由**：AI 未体现工程决策中的取舍与约束，人工补充后更贴合项目实际。

# ADR-003：采用前后端分离 + RESTful API + JWT 身份认证
## 状态
已接受
## 背景
平台支持 Web 端与移动端，需要统一接口、多端同步、安全登录、匿名机制与权限控制。
## 决策
我们决定采用**前后端分离架构**，基于 **RESTful API** 提供服务，并使用 **JWT** 实现身份认证与权限控制。（diff）
## 理由
1. 支持多端数据统一，便于未来扩展移动端
2. 接口清晰，前后端分工明确
3. JWT 无状态、轻量、易于部署（diff）
4. 便于实现树洞匿名、登录权限、管理员权限（diff）
5. 方案成熟、学习成本低、资料丰富
6. 增加 Refresh Token 机制：Access Token 有效期 2h，Refresh Token 有效期 7d，避免频繁登录（diff）
## 后果
- 正面影响：多端适配强、权限灵活、架构清晰
- 负面影响：JWT 无法主动销毁，必须设置合理过期时间
- 需要关注的风险：Token 泄露可能带来越权风险，需配合签名与 HTTPS（diff）
## AI 辅助记录
- **AI 初稿内容摘要**：建议前后端分离与 Token 认证。
- **人工修订内容（diff）**：
  - **明确**：RESTful API 规范
  - **明确**：JWT 选型及其理由
  - **新增**：匿名机制、权限控制、安全风险分析
- **修订理由**：AI 缺少工程实现细节，人工补充后更完整、可落地。

# ADR-004：不采用微服务架构
## 状态
已废弃
## 背景
在架构选型中，对微服务架构进行了可行性评估。微服务具备良好扩展性与模块隔离能力，但需结合团队规模、周期与业务场景进行权衡。
## 决策
我们决定**不采用微服务架构**，维持单体分层架构方案。（diff）
## 理由
1. 微服务需要引入网关、注册发现、分布式事务等额外技术体系，实施成本高
2. 项目业务边界清晰，单体架构已能满足维护性与稳定性需求
3. 4 人团队难以承担微服务运维、联调、故障排查成本（diff）
4. 10 周周期内完成微服务交付风险较高（diff）
5. 校园用户规模与并发量较低，无需微服务级别的扩展能力（diff）
## 后果
- 正面影响：架构简洁、开发高效、交付可控、学习成本低
- 负面影响：不支持模块级独立扩容
- 需要关注的风险：未来业务增长后可能需要重构
## AI 辅助记录
- **AI 初稿内容摘要**：中立介绍微服务优缺点，未给出明确取舍。
- **人工修订内容（diff）**：
  - **明确废弃**微服务架构
  - **新增**：结合团队规模、开发周期、业务规模进行论证
- **修订理由**：AI 未结合实际工程约束，人工修订使决策更合理、可落地。

# ADR-005：是否引入 Redis 做热点数据缓存
## 状态
待讨论
## 背景
树洞首页、热门搭子、用户会话可使用缓存提升性能，但会增加架构与部署复杂度。
## 决策
我们决定**现阶段暂不引入 Redis，以最简架构开发核心功能，后期根据性能测试结果再决定是否接入**。（diff）
## 理由
1. 初期用户量少，MySQL 足以支撑业务
2. 减少外部组件，降低部署与调试成本
3. 不影响核心功能交付
4. 后期可低代价接入
## 后果
- 正面影响：架构极简、开发更快、交付风险更低（diff）
- 负面影响：高并发场景下性能可能不足
- 需要关注的风险：若活跃用户增加，接口响应可能变慢
## AI 辅助记录
- **AI 初稿内容摘要**：建议直接引入 Redis 提升性能。
- **人工修订内容（diff）**：
  - **调整**：改为“暂不引入、后期评估”
  - **新增**：不过早优化，符合课程项目开发节奏
- **修订理由**：AI 偏向技术优化，人工修订更贴合项目实际阶段目标。

# ADR-006：是否引入 WebSocket 实现即时通讯与实时通知
## 状态
已接受（P4 实现）
## 背景
平台包含搭子聊天、恋爱交友聊天、实时通知等功能。HTTP 轮询开销大、延迟高，WebSocket 可实现服务端主动推送，但会增加架构复杂度。
## 决策
我们决定 **P3 阶段预留 WebSocket 接口设计，P4 编码阶段正式实现**。（diff）
## 理由
1. 聊天消息和通知推送对实时性有明确需求（用户期望即时收到消息）
2. P3 详细设计阶段需预留 WebSocket 端点，避免后期接口不兼容
3. P4 编码阶段实现 WebSocket，不影响前三个阶段的交付节奏
4. 当前阶段（P3 及之前）使用 HTTP 轮询作为过渡方案
## 后果
- 正面影响：聊天体验流畅、通知即时、减少不必要的 HTTP 请求
- 负面影响：需维护 WebSocket 连接状态，增加服务端资源开销
- 需要关注的风险：WebSocket 连接断开后的重连机制、消息可靠性
## AI 辅助记录
- **AI 初稿内容摘要**：建议直接使用 WebSocket 实现聊天功能。
- **人工修订内容（diff）**：
  - **调整**：P3 预留设计、P4 正式实现
  - **新增**：HTTP 轮询过渡方案
- **修订理由**：AI 未区分设计阶段和实现阶段，人工修订确保各阶段职责清晰。

## 6. 统一状态枚举定义（全系统共用）

为保障 P1-P3 各阶段术语一致性，定义以下全系统统一状态枚举：

### 6.1 内容状态（适用于树洞动态、搭子需求、交友需求、评论）
| 状态 | 说明 |
|------|------|
| DRAFT | 草稿，用户保存但未提交 |
| PENDING | 待审核，已提交等待自动/人工审核 |
| PUBLISHED | 已发布，审核通过，前台可见 |
| REJECTED | 已驳回，审核不通过，需修改后重新提交 |
| HIDDEN | 已隐藏，用户主动隐藏/到期自动下架 |
| DELETED | 已删除，逻辑删除，不可恢复 |

### 6.2 匹配状态（适用于搭子匹配、恋爱匹配）
| 状态 | 说明 |
|------|------|
| PENDING | 待响应，申请已发出等待对方处理 |
| ACCEPTED | 已接受，双方匹配成功 |
| REJECTED | 已拒绝 |
| CANCELED | 已取消，申请方主动撤销 |
| ENDED | 已结束，搭子/交友关系终止 |

### 6.3 审核结果（适用于审核记录）
| 结果 | 说明 |
|------|------|
| PASSED | 审核通过 |
| REJECTED | 审核驳回 |
| WARNING | 审核警告（内容发布但标记风险） |

### 6.4 反馈处理状态
| 状态 | 说明 |
|------|------|
| PENDING | 待处理 |
| VIEWED | 已查看 |
| PROCESSING | 处理中 |
| RESOLVED | 已解决 |
| CLOSED | 已关闭 |

### 6.5 用户状态
| 状态 | 说明 |
|------|------|
| NORMAL | 正常 |
| WARNED | 已警告 |
| MUTED | 禁言（可浏览不可发布） |
| BANNED | 封禁 |

### 6.6 认证状态
| 状态 | 说明 |
|------|------|
| UNCERTIFIED | 未认证 |
| PENDING | 认证审核中 |
| CERTIFIED | 已认证 |
| REJECTED | 认证驳回 |

## 8. P2→P3 衔接要求

为确保 P3 详细设计与 P2 架构决策一致，P3 设计需遵循以下约束：

1. **状态枚举统一**：P3 类图、API、数据库的 status 字段全部使用 P2 §6 定义的状态值
2. **API 命名规范**：RESTful 资源路径使用 `/api/v1/{module}/{resource}` 层级结构，JSON 字段使用 lowerCamelCase
3. **API 覆盖完整性**：P3 需补充 Love（交友）、Admin（审核）、通知、反馈四大模块的 API 接口
4. **MVP 范围标注**：P3 设计中的 AES 加密、Refresh Token 等属于 P4 实现阶段内容，需在文档中标注"P4 实现"
5. **WebSocket 预留**：P3 的聊天和通知设计需预留 WebSocket 接口（P4 阶段实现），当前阶段使用 HTTP 轮询过渡
